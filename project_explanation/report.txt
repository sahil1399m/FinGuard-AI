What this project actually is
This is a Multi-Agent Anti-Money Laundering (AML) Auditor — a real-time 
financial crime detection system that mimics what actual compliance teams 
at banks like BNY Mellon, JPMorgan, and HSBC use, but built entirely with free tools.
At its core, a fake bank is constantly generating transactions. Every 3 seconds, 
a new payment fires — someone in Mumbai withdraws $9,900, someone in London sends 
$50 to a friend, someone who logged in from Tokyo is now transacting in New York. 
These hit your backend, which passes each one through a chain of AI agents that 
argue, score, and judge whether it's money laundering. The verdict and the agents' 
exact reasoning appear live on a dashboard the instant it's done.
What makes it genuinely sophisticated is that it doesn't just flag things — 
it explains why, it remembers history, it spots patterns across users, 
and it shows the AI agents disagreeing with each other before a judge makes the final call.

Every feature in detail
Feature 1 — Real-time transaction streaming. simulator.py generates a continuous stream of JSON transactions using the Faker library for realistic names, cities, and user IDs. It produces two types: clean transactions (normal amounts, consistent geography) and pre-seeded suspicious ones (impossible travel, structuring amounts, behavioral spikes). It sends one every 3 seconds to your FastAPI backend via HTTP POST. You can adjust the ratio of suspicious-to-clean from a sidebar in the dashboard.
Feature 2 — Geo-Velocity detection (pure math, no LLM). The geo_velocity.py agent calculates the great-circle distance between a user's previous login location and their current transaction location using geopy. It then divides by the time difference to get required travel speed in km/h. Anything above 900 km/h (commercial jet speed) is physically impossible — flagged immediately. No AI call needed, no cost. Example output: "User logged in from London 10 minutes ago. Current transaction is in Mumbai, 7,194 km away. Required speed: 43,164 km/h. Flagged: impossible travel."
Feature 3 — Structuring / Smurfing detection (pure math). The structuring.py agent checks if the transaction amount falls in the $9,000–$9,999 band — the classic "just below the IRS $10,000 mandatory reporting threshold" pattern that money launderers use. It also checks for multiple transactions from the same user in the same band within 24 hours, which is the smurfing pattern. Pure arithmetic, instant, free.
Feature 4 — Behavioral baseline engine (SQLite memory). This is what most student projects don't have. Every transaction is stored in SQLite via aiosqlite. For each user, the system maintains a 30-day rolling average spend and standard deviation. When a new transaction arrives, it calculates a z-score: how many standard deviations above their personal norm is this amount? Anything beyond 3 standard deviations triggers the behavioral agent. This catches the fraud pattern that no rule-based system can — a user whose own behavior has suddenly changed drastically.
Feature 5 — 0–100 Risk Score engine. Instead of binary CLEAN/FLAGGED, every transaction gets a composite risk score. The formula weights contributions from each agent: geo-velocity flag adds up to 40 points, structuring flag adds up to 35, behavioral deviation adds up to 15, and time-of-day anomaly (transactions between 2–5 AM local time) adds up to 10. The dashboard shows this as an animated gauge that sweeps from green to red. Scores below 30 are green, 30–60 are amber, above 60 are red.
Feature 6 — Prosecution vs Defense debate (the showstopper). For any transaction that scores above 40, two LLM-powered agents argue the case using the Groq API (Llama 3.3-70B). The Prosecution agent reads the transaction data and builds the strongest possible case for why it's fraud. The Defense agent reads the same data and argues why it's innocent — maybe the user is a frequent flyer, maybe the amount is a legitimate business payment. A Judge node reads both arguments and issues a final verdict with explicit reasoning. The dashboard shows all three outputs. This is what no other student project does.
Feature 7 — WebSocket push (true real-time). The FastAPI backend uses native WebSocket support to push verdicts to the Streamlit dashboard the instant LangGraph finishes. There's no polling loop, no 2-second delay. The verdict appears live. This is architecturally identical to how real financial systems work — event-driven, push-based.
Feature 8 — Transaction network graph (fraud ring detection). NetworkX builds a live graph where nodes are user IDs and edges are drawn when two users share suspicious co-patterns: same transaction amount within 5 minutes of each other, or transactions that together form a chain ending near $10,000. PyVis renders this as an interactive HTML graph embedded in the dashboard. When three flagged users form a triangle in the graph, that's a money ring. This is the visual that makes the demo unforgettable.
Feature 9 — Streamlit Control Room dashboard. Four panels: top-left shows the live transaction feed as scrolling JSON. Top-right shows the risk gauge and the agent debate (prosecution argument, defense argument, judge verdict) for the most recent flagged transaction. Bottom-left shows a Plotly timeline of risk scores over time. Bottom-right shows the NetworkX fraud ring graph. A sidebar lets you control the simulator speed and suspicious transaction ratio.

How to test it — datasets and scenarios
You don't need a real dataset. Here's exactly how to test each feature:
For the geo-velocity agent, hardcode a transaction where previous_login is {"location": "London, UK", "time": "10 minutes ago"} and current location is Mumbai, IN. It must flag every time. Also test a legitimate traveler — same city as previous login — it must not flag.
For the structuring agent, send amounts of $9,500, $9,750, $9,999, and then $10,001. The first three must flag; the last must not.
For the behavioral baseline engine, first run the simulator for 5 minutes to build history for a user with normal $200–$500 transactions. Then inject a single $45,000 transaction from that same user ID. The z-score should exceed 3 and trigger the behavioral agent without any LLM call.
For the debate protocol, trigger a transaction that is ambiguous — $9,800 from a user in a city they visited last month. Watch the Prosecution and Defense genuinely disagree and see what the Judge decides.
For the fraud ring graph, use the simulator's built-in "ring scenario" mode (you'll write this) — 3 users who each send $3,200 within 90 seconds of each other, totaling just under $10,000. They should appear as a connected triangle in the NetworkX graph.
For end-to-end stress testing, set the simulator to fire every 1 second instead of 3 and run for 10 minutes. Check that the SQLite database has accumulated all records correctly, the WebSocket connection holds without dropping, and the risk score timeline in the dashboard updates without lag.
For a real-world dataset as a bonus, download the publicly available IEEE-CIS Fraud Detection dataset from Kaggle (it's free, no signup needed with Kaggle CLI). Use it to validate that your scoring formula produces high scores for the transactions labeled as fraud in that dataset. This is a genuine evaluation you can cite in the interview.
